# Feature Specification: DroneScout Remote-ID ASM (DApp)

**Feature Branch**: `017-remote-id-dapp`
**Created**: 2026-05-08 · **Revised (SAPIENT)**: 2026-05-30
**Status**: Draft

> **Scope of this revision.** A sensor node is a SAPIENT **ASM**; a consumer node is a SAPIENT **HLDMM**; **015** defines the session, lane binding, registration, tasking, and fan-out mechanics once. This spec describes only **the DroneScout vendor specifics** — its service, classification taxonomy, `rid.*` extension namespace, and the Remote-ID-to-DetectionReport mapping.

> **This DApp is a SAPIENT *translator*, not a seller.** Per 015's transparent-proxy model (015 FR-S90–S97), the SAPIENT-forwarding **Seller Proxy** is generic and defined once in 015. The DroneScout side specified here is the **Neuron-blind modality→SAPIENT translator** that runs *behind* that proxy: it decodes Remote ID and emits standard SAPIENT (`Registration`/`DetectionReport`/`StatusReport`) over a conformant BSI Flex 335 edge, and it ingests SAPIENT (`Task`/`RegistrationAck`). It holds **no Neuron identity, key, wallet, or lane logic** — the Seller Proxy owns the EIP-8004 identity and `node_id`→PeerID binding (015 FR-S94), the lane routing (015 FR-S92), the end-to-end registration handshake and 003 agent-card publication (015 FR-S96), and the admission gates (015 FR-S80–S84). So "the DS240 ASM" below denotes *this translator together with the generic Seller Proxy*; 017 specifies only the translator half.

## Layer

**DApp (vendor ASM)** — composes the **Shared Application Profile 015** and, through it, the Core SDK (001–013). Per Constitution Principle XII (three-tier model). This spec MUST NOT redefine any 015 or Core requirement; it references them. Concretely it defines a **Neuron-blind SAPIENT translator** that runs behind 015's generic **Seller Proxy** (015 FR-S90–S97); all Neuron-side concerns — identity, signing, lane routing, registration handshake, 003 publication, admission — belong to that proxy, not here.

This spec is **vendor-named** (the BlueMark DroneScout sensor), not modality-named: the registration identifies a DroneScout sensor and the Remote-ID modality it provides, rather than a modality in the abstract. Modality and standard semantics (OpenDroneID, ASTM F3411) are input contracts, not redefined here.

A consumer MAY buy this ASM's `rid` service and consume it with **no fusion** (015 FR-S71).

**Reference implementation (informative, not normative; external repository — not part of this source release):**

The `detection.go` *mapping* now follows the reconciliations in §"Mapping"; the SAPIENT protobuf is generated from the vendored DSTL protos (`internal/sapientpb`, regenerated by `scripts/gen-proto.sh`).

**External (input contract):** ASTM F3411-22a (OpenDroneID over BLE/Wi-Fi) — referenced for ODID message-field semantics (FR-R-M05), not redefined; FAA 14 CFR Part 89, EASA 2019/947 + 2019/945. The BlueMark DroneScout DS230/DS240/DS-400 **MQTT envelope** is **documented normatively in §B** (from manufacturer manual §3 p41) so the translation is implementable without the manual or the reference code.

## Purpose

Specify a **BlueMark DroneScout DS240** (and DS-400) sensor node acting as a SAPIENT **ASM** that offers a single Neuron service — **`rid`** — emitting SAPIENT `DetectionReport`s for drones (and their operators) decoded from broadcast Remote ID, carrying the full Remote-ID payload via the `neuron.rid/1` `object_info` extension. The ASM registers, streams, status-reports, and accepts tasking exactly as 015 prescribes; this spec defines only what is DroneScout-specific.

## Out of Scope

- **The SAPIENT session, lane binding, registration handshake, tasking, fan-out, encoding** — all owned by **015**. This spec references them.
- **OpenDroneID message-field layout** (the ASTM F3411 wire format of `BasicID`/`Location`/`SelfID`/`System`/`OperatorID`/`Authentication`) — external normative (F3411); **referenced, not redefined** (FR-R-M05). *(The DS240 MQTT envelope, base64 `UASdata`, wire quirks, and MAC-keyed state-merge that wrap it ARE specified — §B / FR-R-M*. The reference `internal/dronescout`, `internal/odid`, `internal/state` become informative implementations of §B, no longer the only contract.)*
- **Fusion, plot-to-track association, derived affiliation / friend-foe** — fusion-HLDMM concern (**019**); **CoT/TAK output and the static affiliation policy** — display-consumer concern (**018**); ASTERIX/map UI — unassigned/product.
- **The bespoke `RemoteIdFrame` JSON, geofence path-streams, fused-buyer pattern** from the earlier draft — superseded by SAPIENT/015.
- **Sensor inventory** (count, locations, IPs, partner mapping) — deployment configuration, not protocol semantics.

## Clarifications

### Session 2026-05-30

- Q: Envelope? → A: **SAPIENT BSI Flex 335 v2.0 DetectionReport** via 015 (not a bespoke frame).
- Q: How many SAPIENT objects per drone (drone + operator)? → A: **One DetectionReport for the drone.** Operator data is carried as `rid.operator*` `object_info` on that single report, not as a second standalone "UAS Operator" object. The native `associated_detection` is reserved for the case where the operator is independently tracked as its own detection; the default DroneScout path does not emit one. The alternative — a linked second detection — is documented in §"Mapping" note O1.
- Q: `object_id`? → A: SAPIENT v2.0 requires `object_id` to be a **ULID**; the RID serial is not a ULID. The ASM MUST assign a ULID `object_id` (stable per track per its declared `tracking_type` — *refined by CL-R8/FR-R-D02: keyed on `BasicID[0].UASID`, **not** the MAC, with `tracking_type = TRACKING_TYPE_TRACK`*); the RID serial is carried as `rid.uasId` and mirrored to the native `DetectionReport.id` field. (Corrects the prototype, which used the serial as `objectID`.)
- Q: Where do broadcast authentication and kinematics go (no native SAPIENT slot for "authentication"; no native "Track")? → A: Authentication via the 015 extension mechanism under a declared namespace; kinematics via the native `velocity_oneof` (`ENUVelocity`). See §"Mapping".
- Q: Vendor or modality named? → A: **Vendor** (DroneScout DS240).
- Q: What tasking does the `rid` service accept? → A: **Per-session output filters + the mandatory stream STOP/START baseline** — `region_definition` + `class_filter_definition` (plus an altitude-band filter via the `neuron.rid/1` extension once defined — **pending**, CL-R7), and the mandatory `CONTROL_STOP`/`CONTROL_START` controls (015 FR-S29 / FR-R-D09). It accepts **no** `detection_report_rate` decimation (RID is bursty/low-rate, so per-session rate-limiting adds no value). The DS240 is `NODE_TYPE_PASSIVE_RF` (non-pointable, non-mobile, single-mode), so its `TaskDefinition` advertises **no** node-global `CommandType` (`mode_change`/`look_at`/`move_to`/`patrol`/`follow`/`*_threshold`). The 015 FR-S28 "refuse node-global while multiplexed" rule therefore never triggers for `rid`, and multi-HLDMM fan-out is unconditionally faithful.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — DroneScout sells Remote-ID detections to a consumer over SAPIENT (Priority: P1)

A NATed DS240 sensor node registers a `rid` service and, once a consumer connects (per 015), streams SAPIENT `DetectionReport`s for each broadcasting drone over the p2p lane. This is the MVP and the **first end-to-end SAPIENT-over-Neuron dApp**.

**Why this priority**: Without DroneScout emitting fuse-ready SAPIENT detections on the lanes, there is no Remote-ID data product to exercise 015.

**Independent Test**: Run the reference ASM (the `neuron-rid-bridge` renderer behind a 015 ASM runtime) against a recorded DS240 capture (feed-source `replay`) and a stub HLDMM over Profile F; assert `Registration`(service `rid`)→`RegistrationAck`, then `DetectionReport`s on `/sapient/detection/2.0.0` each with a ULID `object_id`, `rid.uasId` = the serial, a populated `location` with an `error` envelope, an `ENUVelocity`, a `classification` of `{UAV, ~0.99}`, and `detection_confidence ≈ 1`.

**Acceptance Scenarios**:
1. **Given** a decoded airborne drone (serial, position, ODID horizontal-accuracy enum 11 → 3 m, vertical enum 4 → 10 m), **When** rendered, **Then** the `DetectionReport` has `location.error.horizontalM = 3`, `verticalM = 10`, `classification[0] = {type:"UAV", confidence:0.99}`, `detection_confidence = 1`, and the on-lane `node_id` resolves to the ASM's EIP-8004-bound identity (bound by the Seller Proxy, 015 FR-S94; the blind translator's local `node_id` UUID is mapped to it).
2. **Given** a drone with `BasicID` but no merged Location, **When** rendered, **Then** an identity-only report is routed to the ASM's advertised behaviour for position-relaxed output (015 FR-S35) — it is NOT emitted on the assembled `/sapient/detection/2.0.0` stream with a missing `location`.
3. **Given** a drone with no `BasicID` / empty UASID, **When** rendered, **Then** no report is emitted (no identity to key on).

### User Story 2 — Consumer discovers DroneScout capabilities from the registry (Priority: P1)

A consumer resolves the DS240's agent-card (the SAPIENT `Registration`, pointer model per 015) from the registry and learns: vendor identity (BlueMark / DroneScout DS240), `NodeType = PASSIVE_RF`, the `rid` service, the UAV taxonomy, coverage (RF reception footprint), the declared `StatusReport` interval, accepted tasks, and the `neuron.rid/1` extension schema — before connecting.

**Acceptance Scenarios**:
1. **Given** a registered DS240, **When** the consumer does `003 LookupRegistration`, **Then** the agent-card's `config_data` carries manufacturer "BlueMark" + model "DroneScout DS240", `node_definition.node_type = NODE_TYPE_PASSIVE_RF`, the `rid` service, and `extensions[]` containing `neuron.rid/1` (id + URI + hash).
2. **Given** the resolved card, **When** a later report carries `rid.*` `object_info`, **Then** the consumer interprets it using only `neuron.rid/1` (015 FR-S40), zero DroneScout-specific code.

### User Story 3 — One drone, operator carried on the same report (Priority: P2)

A drone broadcasting a System message with an operator location yields **one** `DetectionReport` whose `object_info` carries `rid.operatorId`, `rid.operatorLocationType`, and the operator's position fields — not a second fake-aircraft object.

**Acceptance Scenarios**:
1. **Given** a drone with a System operator location, **When** rendered, **Then** exactly one `DetectionReport` is emitted (`classification` UAV), carrying `rid.operatorLatDeg`/`rid.operatorLonDeg`/`rid.operatorAltM`, `rid.operatorLocationType`, and (when present) `rid.operatorId`/`rid.operatorIdType`.
2. **Given** an operator location of exactly `0,0`, **When** rendered, **Then** the operator fields are still emitted at `0,0` (no-suppression; the HLDMM judges usability).
3. **Given** a drone with no System message, **When** rendered, **Then** no operator fields appear (and no operator object).

### User Story 4 — Broadcast authentication forwarded for verification at the HLDMM (Priority: P2)

When a drone broadcasts a Remote-ID signature, the ASM forwards the signature bytes + metadata via the declared extension namespace with `verification = unverified`; the HLDMM (the verification authority per 015 FR-S16) checks it against the signer's registry key.

**Acceptance Scenarios**:
1. **Given** a Message-Set-Signature broadcast, **When** rendered, **Then** the auth extension carries `authType = MessageSetSignature`, `signaturePresent = true`, `verification = unverified`, and the base64 signature bytes.
2. **Given** an unsigned broadcast, **When** rendered, **Then** the auth extension is `{authType: None, signaturePresent: false, verification: unsigned}`.
3. **Given** any broadcast, **When** rendered by the ASM, **Then** `verification` is never `verified`/`failed` — those are the HLDMM's verdict.

### User Story 5 — Feed-source disclosure for TEVV (Priority: P3)

The ASM advertises whether its data plane is `live`, `replay`, `synthetic`, or `placeholder` so a validator (010) can tell operational from fixture evidence.

**Acceptance Scenarios**:
1. **Given** any seller invocation, **When** it emits a SAPIENT `StatusReport` (the sensor health/coverage/mode beat — 015 FR-S31, **distinct** from the Neuron 005 liveness heartbeat), **Then** it advertises `feedSource ∈ {live, replay, synthetic, placeholder}` (absent ⇒ `live`); a non-`live` value marks the run as fixture evidence.

### Edge Cases

- **No identifier** → no report (FR-R-D01).
- **No fix** → identity-only contact routed to the position-relaxed path (015 FR-S35), not emitted on the assembled stream with a null `location`.
- **NaN/Inf** in a decoded value → sanitised to 0 (no other mutation); never dropped (no-suppression).
- **Second BasicID** (serial + CAA registration) → both carried (`rid.uasId`/`rid.uasId2`).
- **Unsigned vs unverified** are distinct; ASM never emits `verified`/`failed`.

## Requirements *(mandatory)*

### Functional Requirements (FR-R-*)

**A. Service & vendor identity**

- **FR-R-A01**: The DS240 ASM MUST register exactly one service named **`rid`** (008), discoverable in the 003 registry per 015 FR-S70. The `rid` service MUST carry an 008 `neuron-commerce` `pricing` object (008 FR-P01/P03); 017 fixes neither the unit nor the amount — both are deployment config (CL-R3), and a **free** deployment sets `amount:"0"`.
- **FR-R-A02**: The agent-card (= SAPIENT `Registration`, 015 FR-S20) MUST set `config_data` `manufacturer = "BlueMark"`, `model = "DroneScout DS240"` (or `"DroneScout DS-400"`), and `node_definition.node_type = NODE_TYPE_PASSIVE_RF` (broadcast Remote-ID is passive RF interception). `capabilities` MUST describe the RF reception footprint as coverage (RID is received, not ranged — no steerable FoV). The single `mode_definition` MUST declare `scan_type = SCAN_TYPE_FIXED` (the RF reception footprint is fixed/omnidirectional — no steerable FoV) and `tracking_type = TRACKING_TYPE_TRACK` (the DS240 persists a stable `object_id` across detections of the same drone — and across broken tracks / BLE MAC rotation — by keying on the broadcast Remote-ID identity, FR-R-D02). A fill-in-the-blanks template is co-located: **`agent-card.example.json`** (illustrative; normative requirements are here + 015 FR-S20). What part of this card goes **on-chain (007)** vs **off-chain (resolved via `agentURI`)** is specified in **015 FR-S24**. The translator **declares this SAPIENT-capability content in its `Registration`**; the **Seller Proxy synthesises and publishes the 003 agent-card from it and fills the Neuron-side blocks** (the 003 `services[]` — `neuron-topic`×3 + `neuron-p2p-exchange` + the 008 `neuron-commerce` service — plus the `identity` binding, per 003/004, 015 FR-S20a) — the blind translator does not publish to 003 itself (015 FR-S96).
- **FR-R-A03**: The agent-card MUST declare the `neuron.rid/1` extension (id + resolvable URI + content hash) in `extensions[]` (015 FR-S40), covering the `rid.*` taxonomy and the authentication block (CL-R1).
- **FR-R-A04**: The classification taxonomy the ASM declares and emits MUST be `UAV` (the craft). Operator data is metadata on the craft report, not a separate declared object class (FR-R-D04).
- **FR-R-A05** *(admission — all three gates per 015 §M / FR-S80–S84)*: This deployment exercises **all three admission gates**, with concrete policy as deployment config (no partner list in this spec, per 015 FR-S81): (a) **network entry** — a **permissioned** 007 Identity Registry whose `AllowlistPolicy` key is held by the coalition operator gates which nodes may register at all (FR-S84); (b) **seller-side** — the DS240 MAY allowlist which consumers may buy the `rid` service via 008 `AdmissionPolicy`, denials surfaced as `AdmissionDenied` (FR-S83); (c) **consumer-side** — an HLDMM MAY allowlist which DS240 sensors it accepts, refusals expressed as SAPIENT `RegistrationAck.acceptance = false` → `RegistrationRejected` (FR-S82). Both peer gates are independent and evaluate the counterpart's EIP-8004-bound identity, never an address. The two registrations in play — **ledger/EIP-8004 registration** (network entry) and **SAPIENT `Registration`** (per-session) — MUST be named distinctly per 015 FR-S23. **All three gates are enforced by the Seller Proxy / 008 / the consumer's `RegistrationAck`, not by the Neuron-blind translator**; a 007/008 denial that the blind translator cannot see surfaces to it as a SAPIENT `Error` (015 FR-S97).

**B. DS240 MQTT ingest — source format** *(the input contract, specified so a bridge is implementable without the reference code. ODID message-field semantics are external (ASTM F3411); everything DS240-specific is normative here. Source: DroneScout DS230/DS240 manual §3 p41.)*

- **FR-R-M01** *(transport)*: The DS230/DS240 publishes RID observations as **JSON over MQTT**. An ASM MUST ingest these — by embedding a broker the sensor publishes to, or by subscribing to the sensor's broker. Topic names are sensor/deployment-defined (reference firmware uses per-sensor upload topics, e.g. `/sensor/<id>/upload`); an ASM MUST NOT derive semantics from the topic — the payload is self-describing (FR-R-M02).
- **FR-R-M02** *(envelope schema)*: A RID **data message** is a JSON object with a numeric `protocol` and a `data` object:

  | JSON key | Type | Meaning |
  |---|---|---|
  | `protocol` | number (`1.0`) | wire version (FR-R-M03) |
  | `data."sensor ID"` | string | publishing sensor identity |
  | `data.RSSI` | int | received signal strength (dBm) |
  | `data.channel` | int | capture channel |
  | `data.timestamp` | int | epoch **milliseconds** |
  | `data."MAC address"` | string | colon-separated radio MAC of the broadcasting UA — the track key (FR-R-M07) |
  | `data.type` | string | `BLE legacy` \| `BLE long range` \| `WiFi NaN` \| `WiFi beacon` |
  | `data.UASdata` | string | base64 of the raw `ODID_UAS_Data` pack (FR-R-M05) |
  | `data.raw` | string (optional) | base64 of the air-interface frame |

  `protocol` is a JSON **number** (not a string); two keys contain **spaces** (`"sensor ID"`, `"MAC address"`). An ASM MUST tolerate unknown extra keys.
- **FR-R-M03** *(protocol gate)*: Only `protocol == 1.0` is supported; other versions MUST be skipped, not errored mid-stream.
- **FR-R-M04** *(message-kind discrimination)*: The sensor also emits non-RID envelopes (`status`, `aircraft`, `location`, `mobile network`). An envelope is a RID source **iff** it carries a `data` object; envelopes without `data` MUST be skipped (not treated as errors).
- **FR-R-M05** *(UAS payload → ODID)*: `data.UASdata` MUST be **base64 (standard)** of the raw, fixed-size **`ODID_UAS_Data`** message pack (914 bytes in the reference firmware). After base64-decode, an ASM MUST parse it per **ASTM F3411 (OpenDroneID)** — external normative; this spec does **not** redefine ODID — into its constituent messages (`BasicID` ×2, `Location`, `SelfID`, `System`, `OperatorID`, `Authentication` incl. multi-page auth). `data.raw`, when present, is base64 of the air-interface frame and MAY feed a position-relaxed/raw service (FR-R-D08 / 015 FR-S35); it is not required for the assembled report.
- **FR-R-M06** *(wire robustness)*: The firmware NUL-terminates and MAY NUL-pad the payload, and in **transmit_mode 2** concatenates several envelopes into one MQTT publish. An ASM MUST strip leading/trailing NUL and whitespace and decode the payload as a **stream of back-to-back JSON values**, yielding each `data` message in wire order. A JSON/base64 error MAY halt the batch but MUST preserve messages already recovered.
- **FR-R-M07** *(state merge + envelope carry-through)*: Drone state MUST be keyed by **`MAC address`**; each data message MUST merge its ODID messages into that MAC's entry (one UA broadcasts `BasicID`/`Location`/`System`/… as separate messages over time). The envelope metadata MUST carry through to the `DetectionReport` (§C): `RSSI` → `signal.amplitude` (FR-R-D03), falling back to `rid.rssiDbm` when no `signal` block is emitted (FR-R-M08); `channel` → `rid.channel` (the radio channel number — **not** a frequency; see FR-R-M08); `timestamp` (ms) → the report time base; `"sensor ID"` → the ASM source association (= the ASM's own `node_id` for a single-sensor DS240; SAPIENT v2.0 has no separate `sourceID` field); `type` → `rid.transport` (SAPIENT `Signal` has no transport field) — the band it implies feeds `signal.centre_frequency` per FR-R-M08; `"MAC address"` → `rid.macAddress` (FR-R-D05). A `DetectionReport` is assembled from the merged per-MAC state, not from a single envelope.
- **FR-R-M08** *(RF `signal` & `centre_frequency` — deterministic conversion)*: SAPIENT's RF `signal` makes `centre_frequency` **mandatory whenever a `signal` block is emitted**. For **WLAN**, the DS240's `channel` is a **real radio channel number** (scan list `/root/wlan_channels.conf`), so `centre_frequency` MUST be computed **deterministically** from `(type, channel)` via the published IEEE 802.11 channel plan — never the raw integer, never per-implementation invention. For **BLE**, the DS240 reports `channel = 0` (it does **not** expose the advertising channel — verified by live capture, CL-R4), so `(BLE, 0)` does not resolve and the `signal` block is omitted (RSSI → `rid.rssiDbm`):

  | `type` | channel → `centre_frequency` (MHz for readability; **the field is reported in Hz** — see below) |
  |---|---|
  | `BLE legacy` / `BLE long range` | DS240 reports `channel = 0` → does **not** resolve → **omit** `signal`, carry RSSI in `rid.rssiDbm` (verified live, CL-R4). *(Were the advertising channel ever exposed: `37`→2402, `38`→2426, `39`→2480.)* |
  | `WiFi NaN` / `WiFi beacon`, ch 1–13 (2.4 GHz) | `2407 + 5·channel` (ch1=2412, ch6=2437, ch11=2462) |
  | `WiFi NaN` / `WiFi beacon`, ch 14 | `2484` (special case) |
  | `WiFi NaN` / `WiFi beacon`, ch ≥ 32 (5 GHz) | `5000 + 5·channel` (ch36=5180, ch149=5745) |

  The channel-number **range disambiguates the Wi-Fi band** (1–14 ⇒ 2.4 GHz, ≥32 ⇒ 5 GHz), so the conversion needs no out-of-band knowledge. The raw `channel` integer MUST also be carried verbatim in `rid.channel` (FR-R-M07). An ASM MUST set `amplitude` from `RSSI`. **`centre_frequency` MUST be reported in Hz (SI)** — the table is in MHz for readability; the emitted value is `MHz × 10⁶` (e.g. ch 6 → `2437000000`). SAPIENT v2.0 carries no unit enum on `Signal`, so SI Hz is the interoperable default (resolved 2026-06-01). If `(type, channel)` does **not** resolve to a frequency in the table (out-of-range / unknown), the ASM MUST **omit** the RF `signal` block (still carrying strength/channel via `rid.*`) rather than emit an `amplitude`-only block or a wrong frequency. *(Supersedes both the earlier draft clause that mapped `channel → centre_frequency` directly and the interim band-nominal rule.)*

**C. Remote-ID → DetectionReport mapping (the DroneScout payload)**

- **FR-R-D01**: The ASM MUST emit, per decoded drone with a usable identity, exactly **one** `DetectionReport` (the craft). No report when `BasicID`/UASID is absent.
- **FR-R-D02** *(object_id & track identity)*: `object_id` MUST be a **ULID** assigned by the ASM and held **stable per drone track**, consistent with the declared `mode_definition.tracking_type = TRACKING_TYPE_TRACK` (registration.proto: *"persist an `object_id` between detections of the same object, even when tracks are broken"*). The **track key MUST be the broadcast Remote-ID identity `BasicID[0].UASID`** (whatever its `idType`) — **not** the MQTT `MAC address`. The MAC is the transient radio/merge key (FR-R-M07) and rotates under BLE MAC randomisation, so a MAC change with an unchanged `UASID` MUST **reuse** the same `object_id` (tracks MUST NOT fragment on MAC rotation); a change in the primary `UASID` is a new track and MUST mint a **new** `object_id`. Because FR-R-D01 emits no report without a usable identity, an `object_id` is never assigned to an identity-less contact. The RID serial/`UASID` (`BasicID[0].UASID`) MUST be carried as `rid.uasId` and SHOULD be mirrored to the native `DetectionReport.id` field (CL-R2).
- **FR-R-D02a** *(report_id — mandatory, fresh ULID)*: Every `DetectionReport` MUST carry a `report_id` that is a **fresh ULID minted per report** — the DSTL proto marks `report_id` both `is_mandatory` and `is_ulid`. It is distinct from the per-track `object_id` (FR-R-D02): `object_id` is stable for the track, `report_id` is unique per emission. A derived/composite form (e.g. `<serial>@<nanos>`, as the reference bridge currently emits) is **non-conformant**; the spec governs. *(ULID = [github.com/ulid/spec](https://github.com/ulid/spec), the same standard `object_id` uses. Found during 016's proto-mandatory audit, 016 CL-J15.)*
- **FR-R-D03** *(native fields)*: The ASM MUST map: geometric position → `location` (`Location`, WGS84, `altitude` geometric) with `location.error` from the ODID horizontal/vertical accuracy enums (metre bounds; omit `error` when both unknown); kinematics (speed, heading, vertical rate) → `velocity_oneof` as **`ENUVelocity`** (E = speed·sin(heading), N = speed·cos(heading), U = vertical rate); a `classification` of `{UAV, confidence}`; a `detection_confidence` (≈1 for a cooperative broadcast); and, optionally, an RF `signal` carrying RSSI→`amplitude` and a `centre_frequency` derived from `(type, channel)` — see **FR-R-M08** (the broadcast transport itself is carried as `rid.transport`, since SAPIENT `Signal` has no transport field; the channel number is *not* a frequency).
- **FR-R-D04** *(operator, one-report model)*: When a System message carries an operator location, the ASM MUST carry the operator on the **same** craft `DetectionReport` via `object_info`: `rid.operatorLocationType`, `rid.operatorLatDeg`/`rid.operatorLonDeg`/`rid.operatorAltM` (passed through untouched incl. `0,0`), and `rid.operatorId`/`rid.operatorIdType` when present. The ASM MUST NOT emit a second standalone operator detection on the default path (note O1).
- **FR-R-D05** *(rid.* extension)*: The ASM MUST carry the remaining decoded Remote-ID payload as `object_info` under the `neuron.rid/1` namespace per **Annex A** (`neuron.rid-1.extension.md` + schema, co-located & normative): second BasicID (`rid.idType2`/`uasId2`/`uaType2`), self-ID (`rid.selfId`/`selfIdType`), Location extras (`rid.status`, `baroAltitudeM`, `heightM`/`heightType`, `timestampSecsThisHour`), accuracy bounds not in `location.error` (`rid.speedAccuracyMps`, `baroAccuracyM`, `timestampAccuracySec`), System regulatory/area fields (`rid.classificationType`, `categoryEU`, `classEU`, `areaCount`/`areaRadiusM`/`areaCeilingM`/`areaFloorM`, `systemTimestampSecsSince2019`), and capture metadata (`rid.macAddress`, `rid.channel`, `rid.transport`, and — only when no native `signal` block is emitted — `rid.rssiDbm`).
- **FR-R-D06** *(no-suppression)*: The ASM MUST emit every value field of a present ODID message even when `0` (faithful to the wire); the ONLY exceptions are accuracy enums where `0`=unknown (omitted) and NaN/Inf (sanitised to `0`). It MUST NOT filter contacts on data quality (every ID type, `0,0` fixes, non-serial IDs all emit). [Carries the no-suppression rule.]
- **FR-R-D07** *(authentication)*: The ASM MUST carry the broadcast-signature status via the declared extension (CL-R1): `authType`, `signaturePresent`, `verification ∈ {unsigned, unverified}` (never `verified`/`failed`), the base64 signature bytes when present, and length/pages/timestamp. Verification is the HLDMM's (015 FR-S16).

**D. Position-relaxed handling**

- **FR-R-D08**: An identity-only contact (no merged Location) MUST NOT be emitted on the assembled `/sapient/detection/2.0.0` stream (where 015 FR-S35 requires `location_oneof`). The DS240 ASM, which is single-service (`rid` = assembled), MUST instead withhold the report until a Location is merged, OR (if a deployment opts in) advertise a separate position-relaxed service per 015 FR-S35. Default: withhold until position is available. [DroneScout normally gets a Location quickly; raw RID is rarely position-less for long.]

**E. Lifecycle, tasking, fan-out, health** — all per **015** (FR-S10–S32). This spec adds no new session mechanics. Tasking the DS240 (region-of-interest via `region_definition`, class via `class_filter_definition`) uses 015 FR-S25/26; the `rid` service accepts **no** `detection_report_rate` decimation. An altitude-band task would use the 015 FR-S26a `object_info` extension filter — **not yet defined for `neuron.rid/1`** (pending, CL-R7); until it is declared, the ASM MUST reject an altitude-band `Task` per FR-S26a.

- **FR-R-D09** *(mandatory stream STOP for the `rid` service — binds 015 FR-S29)*: The DS240 ASM MUST declare `CONTROL_STOP` + `CONTROL_START` as accepted per-session controls in the `rid` service's `mode_definition.task` and honour them per 015 FR-S29. On a `Task{CONTROL_STOP}` it MUST **cease emitting drone `DetectionReport`s** on `/sapient/detection/2.0.0` for that session within its declared cease window and MUST `TaskAck` on the auditable lane; it MUST NOT reject a well-formed STOP, and a STOP from one consumer MUST NOT interrupt any other consumer's stream. `CONTROL_START` resumes emission to that session **without** a new SAPIENT `Registration` (the agent-card/ICD is unchanged). The DS240 keeps decoding RID broadcasts internally while stopped — STOP gates *emission to that consumer*, not the sensor's reception — so resume is immediate. Reports already on the p2p stream are not recalled. If the `rid` service is billed per-`DetectionReport` (an 008/deployment choice, CL-R3), a STOP halts billing to that consumer for the stopped interval; under a free or non-per-report model the STOP simply ceases emission. A consumer of `rid` MUST be able to issue STOP and process the `TaskAck` (015 FR-S29 buyer obligation) and SHOULD STOP before disconnecting.

**F. Feed-source variations (TEVV)**

- **FR-R-E01**: The ASM's data plane MUST be produced from exactly one of `live` (real DS240 capture), `replay` (recorded capture), `synthetic` (deterministic generator), or `placeholder` (valid envelope, deterministic stand-in for an unimplemented decode branch). Default for production = `live`.
- **FR-R-E02**: The ASM MUST advertise its `feedSource` in its SAPIENT `StatusReport` (sensor health/coverage/mode, on the auditable lane) — **not** in the separate Neuron 005 liveness heartbeat; the two are distinct (015 FR-S31). **Carrier:** SAPIENT v2.0 `StatusReport` has **no** `object_info`-style extension slot, so `feedSource` MUST ride as a **native repeated `StatusReport.status[]` entry** — `status_type = STATUS_TYPE_OTHER`, `status_level = STATUS_LEVEL_INFORMATION_STATUS`, `status_value = "neuron.feedSource=<live|replay|synthetic|placeholder>"` — which is strictly-conformant SAPIENT and needs **no** extension declaration; a consumer that does not recognise the namespaced `STATUS_TYPE_OTHER` value ignores it. Absent ⇒ `live`. A validator (010) observing `feedSource ≠ live` MUST treat the run as fixture evidence and disclose the value. (The `status[]` carrier is generic and reusable by any ASM.)

**G. Errors**

- **FR-R-F01**: DroneScout-specific errors MUST use a `NEURON-DAPP-REMOTEID-*` domain (006): at minimum `OdidStateMalformed`, `NoUsableIdentity` (informational; produces no report), `AuthExtensionUnavailable`, and the **ingest** codes (§B) `UnsupportedProtocol` (`protocol ≠ 1.0`, skipped — FR-R-M03), `NotADataMessage` (no `data` object, skipped — FR-R-M04), and `BadEnvelope` (invalid JSON or bad base64 `UASdata`/`raw` — FR-R-M05/M06). The skip-class ingest codes (`UnsupportedProtocol`, `NotADataMessage`) are non-fatal and MUST NOT abort the batch (FR-R-M06). Session/profile errors use the 015 `NEURON-SAPIENT-*` domain.

### Key Entities

- **DroneScout ASM** — a BlueMark DS240/DS-400 node, one SAPIENT identity, one `Registration`, one service `rid`; `NodeType = PASSIVE_RF`.
- **DetectionReport (craft)** — the single per-drone product; ULID `object_id`, `rid.uasId` = serial, UAV classification, optional operator fields.
- **`neuron.rid/1`** — the `object_info` extension namespace (drone identity, self-ID, location extras, accuracy, regulatory/area, capture metadata, operator fields, authentication). Defined by **Annex A**: `neuron.rid-1.extension.md` + `neuron.rid-1.schema.json` (co-located in this spec dir, normative).
- **Decoded Drone (input)** — the MAC-keyed merged `state.Drone` from the upstream decoder (out of scope).

The reference implementation `detection.go` predates the verified DSTL v2.0 protos and the single-report model. This spec reconciles as follows:

| # | Reference impl. (`detection.go`) | This spec (normative) | Rationale |
|---|---|---|---|
| M1 | `objectID = serial` | `object_id` = ASM-assigned **ULID**, keyed on `UASID` not MAC (FR-R-D02); serial → `rid.uasId` (+ native `id`) | v2.0 `object_id` is `is_ulid:true`; MAC rotation must not fragment a track |
| M2 | top-level `Track{speed,heading,vrate}` | native `velocity_oneof` = **`ENUVelocity`** vector | v2.0 has no `Track`; velocity is ENU |

| M4 | top-level `authentication` block | `object_info` under `rid.auth.*` (or `neuron.auth/1`, CL-R1) | v2.0 has no `authentication` field; use the extension mechanism |
| M5 | `Signal{type:RF, rssiDbm, transport}` | native `Signal{amplitude=RSSI, centre_frequency=f(type,channel), …}` (**FR-R-M08** deterministic BLE/802.11 conversion); raw channel → `rid.channel`; transport → `rid.*` | v2.0 `Signal` makes `centre_frequency` mandatory; the envelope `channel` is a real channel number, so derive the exact frequency (reported in Hz, FR-R-M08) via the standard channel plan — never `centre_frequency = channel`, never per-impl invention |

**Note O1 (operator alternative):** if a deployment genuinely tracks the operator as an independent object (e.g. a moving controller), it MAY emit a second `DetectionReport` for the operator linked via native `associated_detection` (`association_type = SIBLING`, referencing the craft's ULID `object_id`). This is the documented alternative to the default one-report model; it MUST NOT use the `OP-`/`OP-OP-` synthetic id scheme.

## Encoding on the wire & inspection view (reference ASM)

Encoding is owned by **015 FR-S50** (Protocol Buffers normative; canonical-JSON projection OPTIONAL, reconcilable field-for-field with the `.proto`). This section records how the DroneScout reference ASM realises it — the DApp-specific concretes, not new policy.

- **Wire (default).** A `DetectionReport` is carried inside a SAPIENT `SapientMessage` envelope (`timestamp` + `node_id` (UUID) + `content.detection_report`) and serialised as **BSI Flex 335 v2.0 Protocol Buffers**, framed with a **4-byte little-endian length prefix** on the conformant SAPIENT connection (the edge to the Seller Proxy, 015 FR-S91) — the framing the DSTL `BSI-Flex-335-v2-Test-Harness` and Apex middleware read and write (SAPIENT ICD §2.1.2; `message_io.py` → `struct.pack("<I", …)`). Go bindings are generated from the vendored DSTL protos.
- **Inspection view.** `--sapient-format=json` emits the OPTIONAL canonical-JSON projection (015 FR-S50) as `protojson` of the **same** `SapientMessage` — so what you inspect is exactly the wire message, never a parallel hand-rolled shape. Default is `protobuf`.
- **Field mapping (on top of the *Mapping — reconciliations* table).** `latitude/longitude/altitude → Location.{y,x,z}` with `coordinate_system = LOCATION_COORDINATE_SYSTEM_LAT_LNG_DEG_M`, `datum = LOCATION_DATUM_WGS84_E`; ODID horizontal accuracy → `Location.{x_error,y_error}`, vertical → `z_error`; the `rid.*` extension → `object_info` (`TrackObjectInfo{type,value}`). **`TrackObjectInfo` has no `units` field**, so `rid.*` units are static-per-key (declared in the `neuron.rid/1` schema) and surfaced only in the JSON view. `signal.centre_frequency` is reported in **Hz** (SI; FR-R-M08's channel plan is tabulated in MHz for readability — emitted value is MHz × 10⁶).
- **`node_id`.** The envelope's mandatory `node_id` UUID is the ASM's SAPIENT identity. The Neuron-blind translator MAY set any local UUID (the reference bridge sets it from `--node-id`); the **Seller Proxy binds the on-lane `node_id` to its EIP-8004-bound PeerID** (015 FR-S94), so what a consumer observes is the crypto-bound network identity regardless of the translator's local value.

**Verification.** A captured frame decodes with stock `protoc --decode=sapient_msg.bsi_flex_335_v2_0.SapientMessage` against the DSTL v2.0 schema into a `SapientMessage` whose `detection_report` carries a ULID `object_id` and a fresh ULID `report_id` (FR-R-D02a), the serial in native `id`, an `ENUVelocity`, a `Signal` with `centre_frequency`, and the `rid.*` `object_info`; the `--sapient-format=json` projection of the same run is field-identical (VR-R-05).

### A.1 Accuracy enum → bound *(reproduced from §5; the numbers most at risk if the reference repo is deleted)*

ODID reports accuracy as enums; map each to its **upper bound**. **Code `0` = unknown ⇒ omit the field** (never emit a false `0` bound).

- **Horizontal** (`location.error.horizontalM`): `1`=18520, `2`=7408, `3`=3704, `4`=1852, `5`=926, `6`=555.6, `7`=185.2, `8`=92.6, `9`=30, `10`=10, `11`=3, `12`=1 (m)
- **Vertical / Barometric** (`location.error.verticalM`, `rid.baroAccuracyM`): `1`=150, `2`=45, `3`=25, `4`=10, `5`=3, `6`=1 (m)
- **Speed** (`rid.speedAccuracyMps`): `1`=10, `2`=3, `3`=1, `4`=0.3 (m/s)
- **Timestamp** (`rid.timestampAccuracySec`): bound = enum × 0.1 s

### A.2 Authentication verification lifecycle *(reproduced from §4.1)*

`verification` reflects **who has checked** the broadcast signature, not just its presence:
- `unsigned` — no signature broadcast (`authType = None`).
- `unverified` — signature present but **this ASM has not checked it** (the ASM is not the verification authority; it forwards the bytes).
- `verified` / `failed` — set by the **HLDMM** after checking the signature against the signer's key in the EIP-8004 registry, bound to the ASM's PeerID (015 FR-S16).

The ASM emits only `unsigned`/`unverified`; the signature bytes are always carried when present so the HLDMM (the verification sink) has them.

### A.3 Value domains

The per-field `rid.*` value domains (`rid.idType`, `rid.uaType`, `rid.status`, `rid.heightType`, `rid.classificationType`, `rid.operatorLocationType`, …) are enumerated in `neuron.rid-1.extension.md` §3 and constrained by `neuron.rid-1.schema.json`. An HLDMM MUST interpret `rid.*` values using only those artifacts (declared in the agent-card `extensions[]`, 015 FR-S40) — never hard-coded vendor knowledge.
